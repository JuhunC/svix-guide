01 개념 이해

웹훅(Webhook)이란 무엇인가?

웹훅은 "어떤 일이 생기면, 그 사실을 상대 서버로 먼저 알려주는 자동 알림"입니다. 내가 계속 물어보지 않아도, 이벤트가 발생한 서비스가 스스로 내 서버의 특정 주소(URL)로 데이터를 담은 요청을 보내줍니다.

조금 더 기술적으로 말하면, 웹훅은 어떤 이벤트가 발생했을 때 서버가 미리 등록된 URL로 보내는 HTTP POST 요청입니다. 그래서 흔히 "역방향 API(reverse API)" 또는 "push API"라고도 부릅니다. 일반적인 API는 내가 필요할 때 상대에게 요청을 보내지만, 웹훅은 반대로 상대가 필요한 순간에 나에게 요청을 보내는 구조이기 때문입니다.

웹훅의 핵심: 이벤트가 발생한 서비스가 먼저 내 서버로 데이터를 밀어(push) 보낸다

일상 속 비유로 이해하기

피자를 주문했다고 생각해 봅시다. 두 가지 방법이 있습니다.

🔁 계속 전화하기 폴링

"제 피자 다 됐나요?"라고 5분마다 가게에 전화합니다. 대부분 "아직요"라는 대답만 듣죠. 전화비와 시간이 낭비됩니다.

🔔 알림 받기 웹훅

가게에 내 번호를 남겨두면, 피자가 완성되는 바로 그 순간 가게가 나에게 문자를 보냅니다. 나는 기다리기만 하면 됩니다.

웹훅이 바로 이 "문자 알림"에 해당합니다. 내가 반복해서 확인할 필요 없이, 이벤트가 생기는 순간 알림이 자동으로 도착합니다.

웹훅은 실제로 어떤 데이터를 보낼까?

웹훅 요청의 본문(body)에는 보통 "무슨 일이 일어났는지"를 설명하는 JSON 데이터가 들어 있습니다. 예를 들어 결제가 완료되었을 때 도착하는 웹훅은 대략 이런 모습입니다.

webhook-payload.json
{
  "type": "invoice.paid",
  "id": "evt_29a8f0c2b1",
  "created": 1721460000,
  "data": {
    "invoice_id": "in_1Qx7Zc",
    "amount": 42000,
    "currency": "KRW",
    "customer_email": "user@example.com",
    "status": "paid"
  }
}

여기서 type은 "어떤 이벤트인지"(청구서 결제 완료), data는 "그 이벤트의 상세 내용"입니다. 내 서버는 이 데이터를 받아 주문 상태 업데이트, 이메일 발송, 재고 차감 같은 후속 작업을 자동으로 처리하면 됩니다.

💡
어디서 많이 쓰일까?

결제(Stripe·Toss), 협업 도구(Slack·Discord 알림), 코드 저장소(GitHub push 이벤트), 배송 추적, CI/CD 파이프라인 등 "무언가 발생하면 즉시 반응해야 하는" 거의 모든 곳에서 웹훅이 쓰입니다.

02 비교

폴링(Polling) vs 웹훅(Webhook)

웹훅의 가치를 제대로 이해하려면, 웹훅이 없을 때 쓰던 방식인 폴링과 비교해 보는 것이 가장 좋습니다.

폴링은 클라이언트가 "새로운 소식 있어?"라고 서버에 주기적으로 반복해서 물어보는 방식입니다. 대부분의 경우 "아직 없어"라는 응답을 받기 때문에, 대다수의 요청이 헛수고가 됩니다. 반면 웹훅은 서버가 "생기면 내가 먼저 알려줄게"라고 약속하고, 실제 이벤트가 생겼을 때 딱 한 번 알림을 보냅니다.

폴링은 수많은 헛된 요청을 만들지만, 웹훅은 이벤트가 생긴 순간에만 통신한다
기준폴링 (Polling)웹훅 (Webhook)
통신 방향내가 → 서버에 계속 요청서버가 → 나에게 알림
실시간성주기만큼 지연 발생거의 즉시 도착
서버 부하불필요한 요청 대량 발생이벤트당 1건만
구현 위치클라이언트가 반복 호출수신용 엔드포인트(URL) 1개
적합한 상황데이터가 자주·규칙적으로 바뀔 때이벤트가 드물고 즉시성이 중요할 때
⚖️
폴링이 항상 나쁜 건 아닙니다

웹훅은 "내 서버가 외부 요청을 받을 수 있어야" 동작합니다. 방화벽 안쪽이거나 외부에 URL을 열 수 없는 환경이라면 폴링이나 메시지 큐(RabbitMQ·Kafka·SQS)가 더 적합할 수 있습니다.

03 동작 원리

웹훅은 어떻게 동작하는가?

웹훅 한 번이 오가는 과정은 크게 등록 → 이벤트 발생 → 전송 → 수신·검증 → 응답 → 처리의 흐름을 따릅니다. 전체 그림을 먼저 본 뒤 단계별로 뜯어보겠습니다.

웹훅 한 건이 처리되는 전체 파이프라인 — 실패하면 발신 측이 자동으로 재시도한다

단계별로 자세히

엔드포인트 등록

내 서버에 웹훅을 받을 URL(예: https://myapp.com/webhook)을 만들고, 외부 서비스의 대시보드에 이 주소를 등록합니다. 이때 서비스는 나에게 서명 비밀키(signing secret)를 발급해 줍니다.

이벤트 발생

사용자가 결제를 완료하거나, 누군가 코드를 push하는 등 주목할 만한 일이 외부 서비스에서 일어납니다.

서명 후 전송

외부 서비스는 이벤트 데이터를 JSON으로 만들고, 비밀키로 서명(signature)을 계산해 헤더에 담은 뒤, 내 URL로 HTTP POST 요청을 보냅니다.

수신 & 검증

내 서버는 요청을 받아 서명이 올바른지, 시간이 유효한지 검증합니다. 검증에 실패하면 위조된 요청일 수 있으므로 거부합니다. (자세한 내용은 05~06장)

빠른 응답

검증에 성공하면 무거운 작업을 하기 전에 먼저 200 OK를 반환합니다. 발신 측은 이 응답을 "잘 받았다"는 신호로 인식합니다.

비동기 처리 & 재시도

실제 처리(이메일 발송 등)는 응답 후 백그라운드에서 진행합니다. 만약 내 서버가 2xx를 돌려주지 못하면, 발신 측은 잠시 후 자동으로 다시 시도합니다.

🔑
기억할 3가지 헤더

대부분의 웹훅에는 webhook-id(메시지 고유 ID), webhook-timestamp(보낸 시각), webhook-signature(서명)라는 세 개의 헤더가 함께 옵니다. 검증은 이 세 값을 이용합니다. 서비스에 따라 svix-id처럼 접두어만 다른 경우도 있습니다.

04 실습 · 수신

웹훅 수신 서버 만들기 (Node.js)

이제 직접 웹훅을 받는 서버를 만들어 봅니다. Node.js와 Express를 사용하고, 검증은 Svix 라이브러리로 간단히 처리합니다.

1단계 — 프로젝트 준비

먼저 웹 서버용 express와 서명 검증용 svix 라이브러리를 설치합니다.

bash · 터미널
npm init -y
npm install express svix

2단계 — 수신 엔드포인트 작성

웹훅을 받을 POST /webhook 엔드포인트를 만듭니다. 여기서 가장 중요한 포인트는 본문을 express.raw()로 받는 것입니다. 그 이유는 바로 아래에서 설명합니다.

server.js
import express from "express";
import { Webhook } from "svix";

const app = express();
const secret = process.env.WEBHOOK_SECRET; // "whsec_..." 형태의 비밀키

// ⚠️ 서명 검증에는 "원본 그대로의" 본문이 필요하다 → express.raw 사용
app.post(
  "/webhook",
  express.raw({ type: "application/json" }),
  (req, res) => {
    const payload = req.body;   // Buffer (가공되지 않은 원본)
    const headers = req.headers;

    const wh = new Webhook(secret);
    let msg;
    try {
      // 검증 성공 시 파싱된 이벤트 객체를 반환, 실패 시 예외를 던짐
      msg = wh.verify(payload, headers);
    } catch (err) {
      return res.status(400).json({ error: "서명 검증 실패" });
    }

    // ✅ 검증 통과 — 여기서 이벤트를 처리한다
    console.log("수신한 이벤트 타입:", msg.type);

    // 무거운 작업 전에 먼저 빠르게 200 응답
    return res.json({ received: true });
  }
);

app.listen(3000, () => console.log("웹훅 서버 실행 중 → http://localhost:3000"));
⚠️
왜 원본(raw) 본문이어야 할까?

서명은 본문의 바이트 하나까지 그대로를 대상으로 계산됩니다. 만약 express.json()으로 JSON을 파싱해 버리면, 공백·키 순서 등이 미세하게 바뀌어 서명이 절대 일치하지 않게 됩니다. 그래서 검증 라우트에서는 반드시 파싱되지 않은 원본을 받아야 합니다.

참고로 예전 스타일인 body-parser를 써도 동일합니다. 원본 본문만 확보하면 됩니다.

JavaScript · body-parser 방식
import bodyParser from "body-parser";

app.post("/webhook", bodyParser.raw({ type: "application/json" }), (req, res) => {
  const payload = req.body;
  const headers = req.headers;

  const wh = new Webhook(secret);
  let msg;
  try {
    msg = wh.verify(payload, headers);
  } catch (err) {
    res.status(400).json({});
  }

  // 메시지로 무언가를 처리...

  res.json({});
});

3단계 — 로컬에서 테스트하기

내 컴퓨터에서 돌아가는 localhost는 외부 서비스가 접근할 수 없습니다. 개발 중에는 ngrok 같은 터널링 도구나 Svix CLI의 listen 기능으로 로컬 서버에 임시 공개 주소를 붙여 테스트할 수 있습니다.

bash · 로컬 터널
# Svix CLI로 로컬 서버(:3000)에 공개 URL 연결
svix listen http://localhost:3000/webhook

# 또는 ngrok 사용
ngrok http 3000
05 보안 · 핵심

서명 검증과 보안

웹훅 URL은 인터넷에 열려 있기 때문에, 사실 누구나 그 주소로 가짜 POST 요청을 보낼 수 있습니다. 그래서 "이 요청이 진짜 그 서비스가 보낸 것"임을 확인하는 서명 검증이 웹훅 보안의 핵심입니다.

발신 서비스는 엔드포인트마다 고유한 비밀키로 각 요청에 서명을 만들어 헤더에 담습니다. 나는 같은 비밀키로 서명을 다시 계산해, 헤더의 값과 일치하는지 확인합니다. 일치한다면 내용이 위조되지 않았고 실제 그 서비스에서 왔다는 뜻입니다.

라이브러리로 검증하기

앞 장에서 본 wh.verify()가 실제로 어떤 값을 받는지 명시적으로 살펴봅시다. 헤더 3종과 원본 본문을 넘기면 끝입니다.

verify.js
import { Webhook } from "svix";

const secret = "whsec_MfKQ9r8GKYqrTwjUPD8ILPZIo2LaLaSw";

// 발신 서버가 함께 보낸 헤더들
const headers = {
  "webhook-id": "msg_p5jXN8AQM9LWM0D4loKWxJek",
  "webhook-timestamp": "1614265330",
  "webhook-signature": "v1,g0hM9SsE+OTPJTGt/tmIKtSyZlE3uFJELVlNIOLJ1OE=",
};
const payload = '{"test": 2432232314}';

const wh = new Webhook(secret);
// 실패 시 예외를 던지고, 성공 시 검증된 내용을 반환한다
const verifiedContent = wh.verify(payload, headers);
webhook-id — 메시지 고유 ID webhook-timestamp — 보낸 시각(Unix) webhook-signature — v1,<base64 서명>

재전송(Replay) 공격 막기

공격자가 유효한 요청을 그대로 가로채 나중에 다시 보내는 것을 재전송 공격이라고 합니다. 서명은 유효하기 때문에 이것만으로는 막기 어렵습니다. 그래서 타임스탬프를 함께 검증합니다.

⏱️
5분 규칙

Svix 라이브러리는 타임스탬프가 현재 시각에서 과거·미래로 5분 이상 벗어난 웹훅을 자동으로 거부합니다. 이 검증이 제대로 동작하려면 서버 시간이 NTP로 정확히 동기화되어 있어야 합니다.

검증 실패는 어떻게 응답할까?

검증에 실패하면 이벤트를 처리하지 말고 즉시 거부해야 합니다. 보통 400 Bad Request로 응답합니다. 반대로 검증에 성공하면, 무거운 작업을 시작하기 전에 먼저 200을 돌려주는 것이 좋습니다. 발신 측은 응답이 늦거나 2xx가 아니면 실패로 간주하고 재시도하기 때문입니다.

06 심화 · 원리

서명 검증의 원리 (수동 구현)

라이브러리가 내부에서 무슨 일을 하는지 알면 검증을 훨씬 깊이 이해할 수 있습니다. 이 장은 원리 이해용입니다 — 실무에서는 검증된 라이브러리를 쓰는 것이 안전합니다.

라이브러리 없이 본 서명 검증: 같은 재료로 서명을 다시 만들어 헤더의 값과 비교한다

단계별 코드

Node.js 내장 crypto 모듈만으로 서명을 재현하는 과정입니다.

manual-verify.js
const crypto = require("crypto");

const secret = "whsec_5WbX5kEWLlfzsGNjH64I8lOOqUB6e8FH";

// ① 서명 대상 문자열 구성: "id.timestamp.body" (점으로 연결)
//    body는 반드시 원본 그대로여야 한다
const signedContent = `${svix_id}.${svix_timestamp}.${body}`;

// ② 비밀키에서 whsec_ 접두어를 뗀 뒤 base64 디코딩
const secretBytes = Buffer.from(secret.split("_")[1], "base64");

// ③ HMAC-SHA256으로 서명을 계산하고 base64로 인코딩
const signature = crypto
  .createHmac("sha256", secretBytes)
  .update(signedContent)
  .digest("base64");

// ④ 헤더의 서명(예: "v1,...")에서 버전 접두어를 떼고
//    crypto.timingSafeEqual 등으로 "상수 시간" 비교 → 타이밍 공격 방지
// ⑤ svix-timestamp가 현재 시각과 5분 이내인지 확인 → 재전송 공격 방지
🧪
검증용 테스트 값

아래 입력을 위 과정에 넣으면 항상 같은 서명이 나와야 합니다. 내가 만든 검증기가 올바른지 확인할 때 유용합니다.

테스트 케이스
secret    = 'whsec_plJ3nmyCDGBKInavdOK15jsl'
payload   = '{"event_type":"ping","data":{"success":true}}'
msg_id    = 'msg_loFOjxBNrRLzqYUf'
timestamp = '1731705121'
// 기대 서명:
'v1,rAvfW3dJ/X/qxhsaXPOyyCGmRKsaKWcsNccKXlIktD0='
🛡️
비교는 반드시 "상수 시간"으로

서명을 ===로 비교하면, 일치하는 글자 수에 따라 응답 시간이 미세하게 달라져 타이밍 공격에 노출될 수 있습니다. Node.js에서는 crypto.timingSafeEqual()을 사용하세요.

07 실습 · 발신

웹훅 보내기(생성하기)

지금까지는 받는 쪽이었습니다. 반대로 내 서비스가 다른 사람에게 웹훅을 보내는 쪽이 될 수도 있습니다. 이때는 "누구에게 보낼지(구독)"와 "서명·재시도" 같은 인프라가 필요한데, Svix 같은 도구가 이를 대신 처리해 줍니다.

인증 & 애플리케이션 생성

먼저 인증 토큰으로 클라이언트를 만들고, 웹훅을 받을 대상들을 묶는 단위인 애플리케이션을 만듭니다.

send-setup.js
import { Svix } from "svix";

const svix = new Svix("AUTH_TOKEN");

// 웹훅 구독자(엔드포인트)들을 담는 애플리케이션 생성
const app = await svix.application.create({ name: "My application" });

이벤트 메시지 전송

이제 이벤트가 발생할 때마다 message.create()로 메시지를 보내면, 그 애플리케이션에 등록된 모든 엔드포인트로 자동 발송됩니다. 서명·재시도·로그는 Svix가 알아서 처리합니다.

send-message.js
const svix = new Svix("AUTH_TOKEN");

await svix.message.create("app_Xzx8bQeOB1D1XEYmAJaRGoj0", {
  eventType: "invoice.paid",                       // 이벤트 종류
  eventId: "evt_Wqb1k73rXprtTm7Qdlr38G",           // (선택) 중복 방지용 ID
  payload: {                                       // 받는 쪽에 전달될 실제 데이터
    id: "invoice_WF7WtCLFFtd8ubcTgboSFNql",
    status: "paid",
    attempt: 2,
  },
});
📮
보내는 쪽이 신경 써야 할 것들

서명 생성, 실패 시 재시도(지수 백오프), 전송 로그와 재전송, 구독 관리(엔드포인트 등록/해제) 등입니다. 직접 구현할 수도 있지만, 규모가 커지면 이런 웹훅 인프라는 전용 서비스에 맡기는 경우가 많습니다.

08 실습 · 대시보드

웹훅 대시보드 만들기

웹훅은 눈에 보이지 않게 오가기 때문에, 무엇이 언제 들어왔고 성공/실패했는지 한눈에 보여주는 대시보드가 있으면 디버깅과 운영이 훨씬 편해집니다.

좋은 웹훅 대시보드는 보통 이런 것들을 보여줍니다: 수신한 이벤트 목록, 각 이벤트의 상태(성공·실패·재시도), 페이로드 본문과 헤더, 그리고 성공률·건수 같은 요약 지표입니다.

1단계 — 수신한 이벤트 기록하기

04장의 수신 서버를 확장해, 검증 결과와 함께 이벤트를 저장합니다. (여기서는 메모리 배열을 쓰지만 실무에서는 데이터베이스를 사용하세요.)

dashboard-server.js
const events = []; // 수신 로그 (실무: DB에 저장)

app.post("/webhook", express.raw({ type: "application/json" }), (req, res) => {
  const wh = new Webhook(secret);
  let msg;
  try {
    msg = wh.verify(req.body, req.headers);
  } catch (err) {
    // 검증 실패도 기록해두면 디버깅에 큰 도움이 된다
    events.unshift({ status: "fail", type: "unknown", at: Date.now() });
    return res.status(400).json({});
  }

  events.unshift({
    id: req.headers["webhook-id"],
    type: msg.type,
    status: "ok",
    at: Date.now(),
    payload: msg.data,
  });

  res.json({ received: true });
});

// 대시보드 화면이 읽어갈 간단한 피드 API
app.get("/api/events", (req, res) => res.json(events.slice(0, 50)));

2단계 — 화면에 실시간으로 그리기

프론트엔드에서는 /api/events를 주기적으로 불러오거나(폴링), SSE·WebSocket으로 실시간으로 받아 목록·지표를 갱신합니다. 집계는 대개 이렇게 계산합니다.

JavaScript · 지표 집계
const total = events.length;
const ok = events.filter((e) => e.status === "ok").length;
const successRate = total ? Math.round((ok / total) * 100) : 0;

// 이벤트 타입별 건수 (막대 차트용)
const byType = {};
for (const e of events) byType[e.type] = (byType[e.type] || 0) + 1;

10 정리 · 체크리스트

베스트 프랙티스 & 체크리스트

마지막으로, 웹훅을 안정적으로 받기 위해 꼭 지켜야 할 핵심 원칙을 체크리스트로 정리했습니다.

  • ✓
    항상 서명을 검증한다검증 없는 엔드포인트는 누구나 가짜 데이터를 보낼 수 있는 열린 문입니다. 예외 없이 모든 웹훅을 검증하세요.
  • ✓
    원본(raw) 본문으로 검증한다JSON을 파싱하기 전 원본 바이트로 서명을 확인합니다. 파싱하면 서명이 깨집니다.
  • ✓
    타임스탬프로 재전송을 막는다현재 시각에서 5분 이상 벗어난 요청은 거부하고, 서버 시간을 NTP로 동기화하세요.
  • ✓
    빠르게 2xx로 응답하고, 처리는 비동기로무거운 작업은 응답 후 백그라운드(큐)에서 처리합니다. 응답이 느리면 발신 측이 재시도합니다.
  • ✓
    멱등성(idempotency)을 보장한다재시도로 같은 이벤트가 두 번 올 수 있습니다. webhook-id를 기록해 중복 처리를 막으세요.
  • ✓
    HTTPS만 사용하고 비밀키는 환경 변수로평문(HTTP)은 가로채기에 취약합니다. 비밀키는 코드에 넣지 말고 환경 변수·시크릿 매니저로 관리하세요.
  • ✓
    실패를 기록하고 관측한다성공뿐 아니라 검증 실패·처리 오류도 로깅해, 대시보드로 상태를 상시 모니터링하세요.
